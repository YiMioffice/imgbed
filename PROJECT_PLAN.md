# 马赫环项目规划

本文档用于后续分块开发和验收。原始产品方向见 `DEVELOPMENT_PREP.md`；本规划在不改变原要求的前提下，按当前代码进度重新拆分为可逐块完成的任务。

## 1. 项目原则

### 1.1 产品定位

- 马赫环不是传统图片托管程序，而是统一静态资源托管系统。
- 第一版必须从底层支持图片、JavaScript、CSS、ZIP、EXE、文档、字体、视频等资源类型。
- 上传、访问、外链、流量、权限、配额、后台配置都必须围绕“静态资源”建模，不允许把图片作为唯一核心模型。

### 1.2 技术方向

- 后端：Go 单体服务。
- 前端：Svelte + Vite。
- 默认数据库：SQLite WAL。
- 默认存储：本机文件系统。
- 扩展存储：S3 兼容存储优先，WebDAV 第二优先。
- 部署目标：本机单进程可运行，后续支持 Docker 和单二进制内嵌前端。

### 1.3 UI 与交互要求

- 界面方向为现代化、克制、极简、高级感。
- 不使用 emoji 图标。
- 图形表达优先使用 SVG、Canvas 或统一线性图标。
- 移动端至少覆盖 375px、768px、1024px、1440px。
- 所有表单元素必须有可访问标签。
- 所有可交互元素必须支持键盘焦点。
- 文案只写产品文案，不写自夸式介绍。

### 1.4 安全与质量要求

- 后台必须鉴权。
- 上传必须走资源类型策略校验。
- ZIP、EXE、JS、HTML、SVG 等高风险资源必须有明确策略和响应头。
- 密码和 API Token 不能明文存储到数据库。
- 删除优先软删除，物理清理由后台任务处理。
- 后端关键逻辑必须有单元测试。
- API 错误结构必须保持统一。
- 每个开发块完成后必须至少运行：
  - `go test ./...`
  - `cd frontend && npm run build`
