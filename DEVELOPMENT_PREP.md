# 马赫环静态资源托管程序开发准备文档

## 1. 项目定位

马赫环是一个现代化、轻量化、易部署的静态资源托管程序。它不是传统图片托管程序，而是面向图片、JavaScript、CSS、ZIP、EXE、文档、字体、视频等静态资源的统一上传、管理、外链访问和策略控制系统。

核心目标是用尽量简单的架构完成高性能资源上传、资源管理、外链访问、用户配额、资源类型策略、后台配置和精选资源展示。图片只是其中一种资源类型，系统底层必须从第一版开始按“静态资源托管”建模，避免后续从图片托管改造造成返工。

设计方向为超现代化极简主义。界面必须简洁、克制、有高级感，不使用 emoji 图标；需要图形表达时优先使用自绘 SVG、Canvas 或统一线性图标。

## 2. 第一版目标

- 提供首页、上传页、资源详情页、探索广场和独立后台。
- 支持登录用户上传和可配置的未登录上传。
- 支持用户组、用户配额、单文件限制、总存储限制、总流量限制。
- 支持按资源类型配置上传权限、单文件大小、单资源月流量、用户组差异化限制。
- 支持本机存储和 S3 兼容存储，预留 WebDAV 与更多对象存储驱动。
- 支持后台管理资源、用户、策略组、存储方式、主页样式和探索广场精选内容。
- 首页展示托管资源量、累计流量、存储占用等统计信息，并加入高级感动画。
- 默认部署方式简单，适合单机运行，也能平滑扩展到 Docker 与对象存储。

## 3. 推荐技术架构

建议采用轻量单体架构：

- 后端：Go
- 前端：Svelte 或 SvelteKit
- 默认数据库：SQLite WAL
- 可选数据库：PostgreSQL
- 默认存储：本机文件系统
- 扩展存储：S3 兼容协议、WebDAV、MinIO、Cloudflare R2、Backblaze B2、阿里 OSS、腾讯 COS
- 部署：单二进制内嵌前端资源，或 Docker Compose

选择理由：

- Go 适合处理上传、下载、流式读写、并发访问和单二进制部署。
- Svelte 打包体积小，适合做高质感但轻量的交互界面。
- SQLite 足够支撑单机静态资源托管，部署成本低；后续需要多人高并发时再切 PostgreSQL。
- 存储层做成统一接口，后续添加新存储驱动不会影响业务层。

## 4. 总体模块

### 4.1 前台

- 首页：品牌展示、上传入口、统计展示、探索广场入口、动态视觉背景。
- 上传页：拖拽上传、粘贴上传、批量上传、上传进度、上传结果复制。
- 资源详情页：资源预览或文件信息、外链地址、Markdown、HTML、BBCode、基础元数据。
- 探索广场：管理员精选资源列表，图片类资源展示预览，其他资源展示文件类型、大小和描述。
- 账户页：昵称、密码、API Token、个人用量、上传历史。

### 4.2 后台

后台使用独立入口，例如 `/admin`，并与普通用户页面区分。后台需要管理员权限。

后台功能：

- 仪表盘：资源总数、存储占用、累计流量、今日上传、用户数量、存储驱动状态。
- 主页设置：主页标题、背景图、元素开关、统计展示开关、动画强度、主题点缀色。
- 用户组设置：游客、普通用户、管理员、自定义用户组。
- 配额设置：总容量、总月流量、每日上传数、默认单文件大小、默认允许文件类型、是否允许外链。
- 资源类型策略：按 `jpg`、`png`、`js`、`css`、`zip`、`exe` 等类型设置上传权限、下载权限、单文件大小、单资源月流量、缓存策略和是否允许游客。
- 策略组管理：管理员在后台配置“用户组 + 资源类型”的规则矩阵，例如游客 `jpg` 单文件每月 1GB 流量、登录用户 `jpg` 单文件每月 10GB 流量、游客禁止上传 `zip`。
- 未登录上传：开关、游客策略组、IP 限流、验证码策略。
- 存储设置：本机、S3、WebDAV 等存储配置与健康检查。
- 账户管理：管理员资料、昵称、密码、头像、API Token。
- 资源管理：搜索、筛选、预览、下载、删除、批量删除、软删除、恢复或清理。
- 用户管理：按 UID 查询，编辑用户昵称、状态、用户组、容量、流量、封禁状态。
- 探索广场管理：选择精选资源、排序、标题、描述、上下架。

## 5. 上传与访问流程

上传流程：

1. 接收文件流。
2. 校验登录态或游客上传权限。
3. 识别资源类型：结合扩展名、MIME、文件头嗅探结果，得到归一化资源类型。
4. 解析策略：根据用户组、资源类型、上传入口和当前时间窗口匹配策略规则。
5. 检查用户组配额、资源类型权限、单文件大小、每日上传次数和限流策略。
6. 对图片类资源校验图片解码结果；对 ZIP、EXE、JS 等非图片资源只做文件类型、安全和策略校验。
7. 计算文件 hash，用于重复资源判断和存储路径生成。
8. 图片类资源可选清理 EXIF 元数据。
9. 写入目标存储驱动。
10. 图片类资源生成缩略图和图片元数据；其他资源生成文件元数据。
11. 写入数据库记录。
12. 返回直链、Markdown、HTML、BBCode 和资源 ID。

访问流程：

1. 解析资源 ID 或文件路径。
2. 检查资源状态、公开权限、私有签名和资源类型访问策略。
3. 按资源所属用户组、资源类型和时间窗口检查单资源流量限制。
4. 记录访问流量和访问次数。
5. 从本机或对象存储读取文件。
6. 返回缓存头、内容类型、下载策略和资源内容。

## 6. 存储抽象

存储层建议定义统一接口：

- `Put(ctx, objectKey, reader, size, contentType)`
- `Get(ctx, objectKey)`
- `Delete(ctx, objectKey)`
- `Exists(ctx, objectKey)`
- `Stat(ctx, objectKey)`
- `PublicURL(ctx, objectKey)`
- `HealthCheck(ctx)`

第一阶段实现：

- LocalStorage：本机目录存储，适合默认部署。
- S3Storage：S3 兼容协议，兼容 MinIO、Cloudflare R2 等。

第二阶段实现：

- WebDAVStorage
- AliOSSStorage
- TencentCOSStorage
- BackblazeB2Storage

## 7. 策略组与资源类型流量限制

策略系统是马赫环的基础能力，不能只写成简单的“允许上传文件类型”。第一版应建立独立的策略解析层，后续后台只是在管理这些策略数据。

策略维度：

- 用户组：游客、普通用户、管理员、自定义用户组。
- 资源类型：图片、脚本、样式、压缩包、可执行文件、文档、字体、视频、其他。
- 扩展名：`jpg`、`jpeg`、`png`、`gif`、`webp`、`svg`、`js`、`css`、`zip`、`exe` 等。
- MIME 类型：用于辅助识别和安全校验。
- 时间窗口：每日、每月、永久累计。
- 作用对象：单个资源、单个用户、单个 IP、整个用户组。

每条策略规则建议包含：

- 是否允许上传。
- 是否允许下载或外链访问。
- 单文件大小上限。
- 单资源月流量上限。
- 单用户该类型资源月流量上限。
- 单用户该类型资源总容量上限。
- 每日上传次数上限。
- 是否需要登录。
- 是否需要管理员审核。
- 是否强制私有链接。
- 是否允许设置公开访问。
- 缓存策略，例如 `Cache-Control`。
- 下载策略，例如直接预览或强制下载。

策略示例：

```text
游客 + jpg:
  allow_upload: true
  max_file_size: 10MB
  monthly_traffic_per_resource: 1GB

登录用户 + jpg:
  allow_upload: true
  max_file_size: 50MB
  monthly_traffic_per_resource: 10GB

游客 + zip:
  allow_upload: false

登录用户 + zip:
  allow_upload: true
  max_file_size: 200MB
  monthly_traffic_per_resource: 5GB
  download_disposition: attachment

普通用户 + exe:
  allow_upload: false

管理员 + exe:
  allow_upload: true
  max_file_size: 500MB
  monthly_traffic_per_resource: 20GB
  download_disposition: attachment
```

后台策略组管理要求：

- 管理员可以创建、复制、启用、停用策略组。
- 管理员可以按用户组和资源类型编辑策略矩阵。
- 策略编辑界面需要显示默认策略、覆盖规则和最终生效规则。
- 修改策略后，新上传资源立即按新策略校验；已有资源访问也按最新访问策略与流量限制执行。
- 需要提供“测试策略”能力：输入用户组、文件名、MIME、文件大小，后台返回是否允许上传及原因。

实现建议：

- 后端提供 `PolicyResolver`，统一解析用户组、资源类型、文件大小、访问动作。
- 上传、下载、后台预览都必须调用同一个策略解析层。
- 流量统计按资源 ID、资源类型、用户 ID、时间窗口聚合，避免每次访问都扫描原始日志。
- 策略规则保存在数据库中，配置 JSON 只做初始化默认值，不作为唯一数据源。

## 8. 数据模型草案

核心表：

- `users`：用户 ID、用户名、昵称、邮箱、密码 hash、头像、状态、用户组、创建时间。
- `user_groups`：组 ID、名称、总容量、默认月流量、默认单文件限制、每日上传限制、权限开关。
- `resources`：资源 ID、用户 ID、存储驱动、对象 key、原文件名、扩展名、资源类型、大小、MIME、hash、状态、创建时间。
- `resource_metadata`：资源 ID、图片宽高、图片 EXIF 状态、文件头摘要、下载策略、缓存策略等扩展元数据。
- `resource_variants`：缩略图、预览图、WebP 或 AVIF 版本，仅图片类资源需要。
- `resource_type_policies`：用户组 ID、资源类型、扩展名、上传权限、访问权限、大小限制、流量限制、时间窗口。
- `resource_traffic_windows`：资源 ID、用户 ID、资源类型、窗口类型、窗口开始时间、已用流量、访问次数。
- `storage_configs`：存储类型、配置 JSON、启用状态、默认状态、健康检查结果。
- `site_settings`：站点名称、主页配置、主题配置、上传策略。
- `featured_resources`：精选资源 ID、标题、描述、排序、状态、创建时间。
- `traffic_logs`：资源 ID、用户 ID、访问字节数、访问来源、时间。
- `api_tokens`：用户 ID、token hash、名称、权限、过期时间。

## 9. 首页视觉方向

品牌名：马赫环。

视觉关键词：

- 超现代化极简主义
- 高反差
- 细线条
- 明暗边界
- 扫描感
- 低噪声动效
- 高级感数据展示

建议表达：

- 使用自绘 SVG 或 Canvas 制作“马赫环”视觉隐喻，例如明暗带、边界光晕、细线扫描。
- 首页不要做传统营销页，首屏应直接提供上传入口和核心统计。
- 动画使用 `transform` 与 `opacity`，避免影响布局稳定。
- 必须支持 `prefers-reduced-motion`，用户关闭动态效果时降级为静态背景。
- 不使用 emoji 图标；图标统一为自绘 SVG 或同一套线性 SVG。

首页可展示：

- 托管资源量
- 累计存储占用
- 累计访问流量
- 今日上传量
- 按类型资源数量，例如图片、脚本、压缩包
- 探索广场精选入口
- 上传入口

## 10. 探索广场

探索广场第一版建议做成管理员精选资源广场，而不是开放社区。图片类资源可以直接展示视觉预览，ZIP、JS、EXE 等其他资源展示资源卡片、类型、大小、描述和安全提示。

原因：

- 可控性更高。
- 审核成本低。
- 不需要第一版实现点赞、评论、收藏等复杂功能。
- 更符合静态资源托管工具的轻量定位。

后台能力：

- 从资源管理中选择资源加入精选。
- 设置标题、描述、排序和展示状态。
- 支持首页只展示部分精选，点击后进入完整探索广场。
- 支持下架精选但不删除原资源。

后续可扩展：

- 标签
- 收藏
- 点赞
- 公开用户主页
- 审核队列

## 11. 权限与安全

必须实现：

- 管理员、普通用户、游客三类基础权限。
- 后台访问必须鉴权。
- 上传接口限流。
- 文件类型白名单。
- 资源类型策略校验。
- 图片类资源解码校验。
- ZIP、EXE、JS 等资源必须按策略组严格控制游客和普通用户权限。
- 路径穿越防护。
- SVG 默认禁止上传，或经过严格清洗后才允许。
- 私有资源使用签名链接。
- 密码使用安全 hash，例如 Argon2id 或 bcrypt。
- API Token 只保存 hash，不保存明文。
- 删除资源优先软删除，后台清理任务再物理删除。

建议实现：

- 登录失败次数限制。
- 操作审计日志。
- 后台危险操作二次确认。
- CSRF 防护或 SameSite Cookie 策略。
- 对象存储凭据加密保存。
- EXE 等高风险资源默认禁止游客上传，普通用户也建议默认关闭。
- JS、HTML、SVG 等可能触发浏览器执行的资源需要明确 `Content-Type`、`Content-Disposition` 和安全响应头策略，避免被当作站内脚本执行。

## 12. 配置与部署

默认目录结构建议：

```text
data/
  machring.db
  uploads/
  previews/
config.yaml
machring
```

默认部署方式：

- 单二进制运行。
- 首次启动进入安装向导。
- 安装向导创建管理员账号、站点名称和默认存储方式。
- 默认使用 SQLite 与本机存储。

Docker 部署：

```text
services:
  machring:
    image: machring:latest
    ports:
      - "8080:8080"
    volumes:
      - ./data:/app/data
      - ./config.yaml:/app/config.yaml
```

## 14. 质量门槛

- 后端关键逻辑需要单元测试：上传校验、配额判断、存储接口、权限判断。
- API 需要统一错误结构。
- 前端需要移动端适配，至少覆盖 375px、768px、1024px、1440px。
- 所有交互元素需要可键盘访问。
- 文案不使用 emoji。
- 首页动效不能造成明显卡顿。
- 资源上传失败必须给出明确原因，尤其是策略组拒绝上传时需要返回命中的规则原因。
- 删除、封禁、修改存储配置等危险操作需要确认。

### 第一步：后台与策略系统

目标：先把“管理员能配置策略组”做成系统核心，而不是写死在代码里。

后端任务：

- 实现管理员初始化流程：首次启动创建管理员账号。
- 实现登录接口、退出接口、当前用户接口。
- 实现会话或 Token 鉴权中间件。

- 将当前写死的默认策略迁移为数据库默认种子数据。
- 实现策略 CRUD API：创建、读取、更新、删除、启用、停用、复制。
- 实现策略测试 API：输入用户组、文件名、MIME、文件大小，返回允许/拒绝、命中规则和原因。
- 上传接口改为读取数据库策略，而不是只读取内存默认策略。

前端任务：

- 建立后台基础布局：侧边栏、顶部栏、内容区。
- 增加后台登录页。

### 第二步：资源托管核心闭环

目标：让系统从脚手架变成可以真实使用的静态资源托管程序。

后端任务：

- 接入 SQLite WAL。
- 建立数据库迁移机制。
- 建立核心表：`users`、`user_groups`、`resources`、`resource_metadata`、`resource_type_policies`、`resource_traffic_windows`、`traffic_logs`、`site_settings`。
- 上传成功后写入资源记录。
- 实现资源访问接口，按资源 ID 或对象 key 返回文件。
- 实现真实流量统计：访问时记录字节数，并聚合到当前月窗口。
- 实现单资源月流量限制：超过策略限制后拒绝访问或返回可配置错误页。
- 实现用户用量统计：总容量、资源数量、按类型占用、月流量。
- 实现资源列表、资源详情、删除、软删除、恢复和物理清理任务。
- 实现图片类资源基础元数据读取：宽高、MIME、大小、hash。

前端任务：

- 完善上传页：拖拽上传、文件选择、上传进度、上传结果。
- 上传成功后展示直链、Markdown、HTML、BBCode。
- 实现资源管理页：列表、筛选、搜索、预览、删除。
- 实现资源详情页：元数据、外链、访问量、流量用量。
- 首页流量图接入后端统计，不再使用静态示例数据。
- 首页上传按钮跳转到完整上传页。
- 登录后显示账户入口，未登录显示登录按钮。

验收标准：

- 用户可以上传图片、JS、ZIP 等被策略允许的资源。
- 游客上传 `zip` 会被拒绝，并显示明确原因。
- 上传成功的资源可以通过外链访问。
- 单资源月流量达到限制后，访问被阻止或降级处理。
- 后台资源管理可以删除和恢复资源。
- 首页流量图展示真实统计数据。
